# Git 工作流说明

## 一、目标

本项目的 Git 工作流不只是管理代码版本，还承担以下作用：

- 记录每天开发了什么
- 保留项目演进轨迹
- 支撑日报和阶段总结
- 便于后续复盘和面试讲述

## 二、基本原则

- 每天至少有一次代码提交
- 每天开发结束后更新日报
- 功能完成后优先自测，再提交代码
- 重大节点及时推送到 GitHub
- 提交信息清晰可读，能看出当次改动目的

## 三、推荐日常流程

每天建议按以下节奏推进：

1. 明确当天目标
2. 实现代码或文档改动
3. 完成本地自测
4. 更新 `docs/daily/YYYY-MM-DD.md`
5. 执行 Git 提交
6. 推送到 GitHub

## 四、提交信息规范

推荐使用以下前缀：

- `feat`：新增功能
- `fix`：修复问题
- `docs`：文档更新
- `refactor`：重构
- `chore`：工程或配置类调整

示例：

- `feat: 完成后台登录模块`
- `feat: 新增文章分类管理`
- `fix: 修复登录态校验问题`
- `docs: 更新项目需求说明书`
- `docs: 更新今日日报`

## 五、日报机制

日报目录：

- `docs/daily/`

文件命名：

- `YYYY-MM-DD.md`

日报建议结构：

- 今日目标
- 实际完成
- 关键改动
- 自测结果
- 遇到问题
- 明日计划

作用：

- 清楚记录每天做了什么
- 便于回顾学习过程
- 便于整理面试表达素材

## 六、阶段性总结建议

每完成一个阶段，建议补一份阶段总结，内容包括：

- 本阶段目标
- 实际完成情况
- 关键技术点
- 遇到的问题与解决思路
- 下一阶段计划

可选放置位置：

- `docs/phase-summary/`

## 七、与 Codex 的协作方式

后续默认协作方式如下：

- 你负责给出当天目标或当前模块目标
- Codex 负责实现、联调、自测、总结建议
- 若远程仓库和认证已配置，可继续执行提交和推送相关操作

## 八、推荐起步提交顺序

项目启动阶段建议按下面顺序提交：

1. `docs: 初始化项目需求与迭代计划`
2. `docs: 新增架构与数据库设计文档`
3. `chore: 初始化项目协作规则`
4. `chore: 初始化前后端工程骨架`

## 九、推荐分支策略

项目初期建议保持简单：

- 直接在主分支推进即可
- 如果后续功能变多，再引入功能分支

建议：

- 学习型项目早期优先减少流程复杂度
- 把重点放在持续交付和持续沉淀上
