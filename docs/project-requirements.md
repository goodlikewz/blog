# 项目需求说明书

## 一、项目基本信息

### 1.1 项目名称

学习型技术博客系统

### 1.2 项目定位

本项目是一个以 Java 学习为主线、以个人博客为载体、以知识沉淀和面试展示为结果的学习型技术项目。

它不仅是一个可运行的博客系统，也是一套长期的学习与成长工程，承担以下 4 个目标：

- 搭建一个可运行、可上线、可持续扩展的个人技术博客
- 在开发过程中系统练习 Java 后端项目能力
- 沉淀 Java、Spring Boot、Redis、RocketMQ 等学习内容，形成长期知识库
- 形成一个适合面试展示和项目讲述的个人作品
- 建立一个可持续输出学习总结、场景题拆解、项目复盘文章的内容平台

### 1.3 项目背景

在学习 Java 技术栈的过程中，单纯刷题或零散写 Demo 很难形成完整的工程经验，也不利于长期沉淀技术内容。

因此需要一个既有真实业务场景、又有持续写作价值、同时还能支撑技术扩展的项目载体。博客系统具备以下优势：

- 业务模型清晰，适合作为完整项目入门
- 包含用户、文章、分类、标签、搜索、统计等典型模块
- 可持续产出内容，形成知识积累
- 便于后续引入 Redis、MQ、搜索、微服务等中高级技术
- 能够在面试中同时展示项目能力与技术成长轨迹

## 二、项目建设目标

### 2.1 业务目标

- 提供一个用于发布、管理、展示技术文章的个人博客平台
- 支持后台维护文章、分类、标签和站点基础信息
- 支持访客浏览、筛选和搜索已发布文章

### 2.2 学习目标

- 通过真实项目练习 Spring Boot 项目搭建与分层设计
- 练习 MyBatis-Plus、MySQL、JWT、接口设计、参数校验、异常处理等基础能力
- 在后续迭代中逐步接入 Redis、RocketMQ、Elasticsearch、Spring Cloud 等技术

### 2.3 成长目标

- 将 Java 学习内容沉淀为结构化文章
- 建立持续输出和总结的习惯
- 形成可复盘、可追踪、可展示的学习记录
- 形成“阶段总结 + 专题总结 + 项目总结”三类长期内容资产

### 2.4 面试目标

- 形成一个能完整讲述的个人项目
- 项目中体现真实需求、架构设计、模块拆分、技术选型、优化思路和扩展路径
- 建立一套面向场景面试的问题沉淀机制，而不只是整理概念题
- 能基于项目内容回答常见面试问题，例如：
  - 为什么这样设计表结构
  - 为什么登录采用 JWT
  - 如何实现文章搜索
  - Redis 如何接入热门文章
  - 后续如何做微服务拆分

## 三、用户角色与使用场景

### 3.1 游客

角色说明：

- 无需登录，作为博客访问者浏览内容

主要场景：

- 浏览首页
- 查看文章列表
- 查看文章详情
- 根据分类查看文章
- 根据标签查看文章
- 通过关键字搜索文章

### 3.2 管理员

角色说明：

- 需要登录后台，负责维护整个博客内容

主要场景：

- 登录后台
- 新建文章
- 编辑文章
- 删除文章
- 保存草稿
- 发布文章
- 下线文章
- 维护分类
- 维护标签
- 查看基础统计信息
- 维护系统基础配置

### 3.3 博客作者 / 学习者

角色说明：

- 既是系统管理员，也是内容创作者和学习记录者

主要场景：

- 输出学习总结文章
- 记录面试复盘与场景题拆解
- 整理项目开发过程中的设计思路和踩坑记录
- 建立技术专题和学习路线

## 四、项目范围

### 4.1 第一阶段范围

第一阶段只做核心可用功能，目标是尽快形成一个可发布文章、可展示内容、可后台维护的博客系统。

包含：

- 前台首页
- 学习路线导航
- 后台登录
- 文章管理
- 分类管理
- 标签管理
- Markdown 编辑与展示
- 搜索功能
- 基础统计功能
- 学习总结内容专区
- 项目实战内容专区
- 面试复盘内容专区

### 4.2 第二阶段范围

在第一阶段基础上增强：

- Redis 缓存
- 点赞功能
- 评论功能
- 图片上传
- 热门文章
- 操作日志

### 4.3 第三阶段范围

作为后续技术升级项目扩展：

- RocketMQ 异步处理
- Spring Cloud 微服务拆分
- Gateway 网关
- Nacos 注册中心
- Elasticsearch 全文检索
- Docker 部署

- 学习总结

- 标题
- 摘要
- 分类
- 标签
- 发布时间

- 面试复盘

#### 5.1.5 标签页

- 标签

- 系列名称或专题标识

功能说明：

功能点：

#### 5.2.4 标签管理

功能说明：

- 维护文章标签

功能点：

- 新增标签
- 编辑标签
- 删除标签
- 标签列表查看

#### 5.2.5 统计管理

功能说明：

- 后台查看博客基础统计信息

## 七、非功能需求

### 7.1 可维护性

- 项目结构清晰，便于分模块开发和后续扩展
- 后端采用规范分层，适合学习和维护
- 文档、日报、SQL 脚本持续同步更新

### 7.2 可扩展性

- 第一阶段采用单体架构，但目录结构要为后续扩展保留空间
- 数据表设计尽量兼顾后续统计、评论、缓存等功能
- 搜索模块设计时预留后续接入 Elasticsearch 的思路

### 7.3 可演示性

- 每个阶段都应形成可运行、可展示的成果
- 项目过程可通过 Git 提交和日报追踪
- 项目应能支持后续面试讲述

### 7.4 可持续沉淀

- 博客本身应成为学习记录载体
- 项目开发过程应同步积累技术总结和排障记录
- 上线时应具备基础文章内容，避免只有系统壳子没有内容
- 应形成学习总结、场景题拆解、项目复盘三类核心内容资产

## 十一、阶段性交付目标

### 11.1 第一阶段交付目标

形成一个最小可用版本，具备以下能力：

- 管理员登录后台
- 维护分类、标签、文章
- 发布文章并在前台展示
- 支持基础搜索
- 支持学习路线和内容专题入口
- 至少具备一批可展示的基础文章内容

### 11.2 第二阶段交付目标

形成一个更完整、更接近上线状态的博客系统：

- 具备基础运营能力
- 支持图片上传、阅读量、热门文章
- 具备基础缓存和交互能力

### 11.3 第三阶段交付目标

形成一个具备技术升级价值的学习型项目：

- 具备中间件和架构升级实践
- 可展示更强的工程能力和系统设计思路

## 十二、验收标准

### 12.1 第一阶段验收标准

- 前后端工程可启动
- 数据库可初始化
- 管理员可登录后台
- 分类、标签、文章管理可正常使用
- 前台可浏览已发布文章
- Markdown 渲染正常
- 搜索可用
- 首页能体现学习路线、学习总结、面试复盘、项目实战等内容入口
- 仓库中已有首批文章选题或初始化内容规划

### 12.2 项目过程验收标准

- 文档持续更新
- SQL 脚本可追踪
- 日报可追踪每日开发内容
- Git 提交记录清晰
- 每个阶段都有可演示成果
