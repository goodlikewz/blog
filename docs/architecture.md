# 项目架构说明

## 一、架构目标

本项目的架构设计需要同时满足以下目标：

- 支撑个人博客系统的基础业务运行
- 便于学习和实践 Java 后端开发能力
- 结构清晰，适合长期维护和持续迭代
- 为后续接入 Redis、RocketMQ、Elasticsearch、Spring Cloud 预留扩展空间
- 便于在面试中讲清楚项目结构、模块边界和演进路径
- 支撑学习总结、场景面试、项目复盘等内容体系长期沉淀

## 二、总体架构

本项目采用前后端分离架构。

- 前端：负责页面展示、用户交互、后台管理界面、登录态维护
- 后端：负责业务处理、数据存储、接口提供、登录认证、文章内容管理
- 数据库：负责存储用户、文章、分类、标签、配置等核心数据

第一阶段优先采用单体应用方式，降低复杂度，尽快完成核心闭环。

除了系统功能，本项目还包含一条内容主线：

- 通过内容模型支撑学习路线、学习总结、场景面试、项目实战 4 类内容聚合

## 三、技术选型

### 3.1 后端技术

- Spring Boot：用于快速搭建后端服务
- MyBatis-Plus：用于简化数据访问层开发
- MySQL：用于存储业务数据
- JWT：用于后台登录认证
- Redis：第二阶段用于缓存和热点数据处理

### 3.2 前端技术

- Vue 3：用于构建前端页面
- Vite：用于前端工程化构建
- Element Plus：用于后台管理界面组件
- Markdown 渲染组件：用于文章内容展示
- 代码高亮组件：用于展示技术文章代码块

## 四、模块划分

### 4.1 后端模块

建议目录结构：

```text
blog-server
├── controller
│   ├── admin
│   └── portal
├── service
├── service.impl
├── mapper
├── entity
├── dto
├── vo
├── config
├── common
├── utils
└── BlogApplication.java
```

模块职责：

- `controller.admin`：后台管理接口
- `controller.portal`：前台展示接口
- `service`：业务接口定义
- `service.impl`：业务实现
- `mapper`：数据库访问层
- `entity`：数据库实体对象
- `dto`：请求参数封装
- `vo`：响应结果封装
- `config`：配置类
- `common`：统一返回、异常、常量等公共内容
- `utils`：工具类

### 4.2 前端模块

建议目录结构：

```text
blog-web
├── src
│   ├── api
│   ├── views
│   │   ├── portal
│   │   └── admin
│   ├── components
│   ├── router
│   ├── store
│   ├── utils
│   └── main.js
```

模块职责：

- `api`：统一封装接口请求
- `views/portal`：前台页面
- `views/admin`：后台页面
- `components`：公共组件
- `router`：路由配置与登录态控制
- `store`：状态管理
- `utils`：工具方法

### 4.3 内容域模块

建议在功能设计上把内容域作为重点，而不是只把文章当作简单文本。

内容域建议包含：

- 技术分类
- 标签
- 文章类型
- 学习路线
- 专题聚合

其中第一阶段最少应支持：

- 分类
- 标签
- 文章类型
- 首页专题聚合

推荐的文章类型：

- 学习笔记
- 学习总结
- 项目实战
- 面试复盘
- 场景题拆解
- 踩坑记录

## 五、核心业务流

### 5.1 后台登录流程

1. 管理员输入用户名和密码
2. 前端调用登录接口
3. 后端校验账号和密码
4. 校验通过后签发 JWT Token
5. 前端保存 Token
6. 后续后台请求携带 Token
7. 后端统一校验 Token 合法性

### 5.2 文章发布流程

1. 管理员登录后台
2. 进入文章编辑页
3. 填写标题、摘要、内容、分类、标签等信息
4. 保存为草稿或直接发布
5. 后端写入文章表及文章标签关联表
6. 前台文章列表展示已发布文章

### 5.3 前台浏览流程

1. 游客访问首页或文章列表页
2. 前端调用文章列表接口
3. 后端查询已发布文章并返回分页数据
4. 游客点击文章进入详情页
5. 前端调用文章详情接口
6. 后端返回文章内容并更新阅读量

### 5.4 内容沉淀流程

1. 作者在学习或项目实践中形成问题或总结
2. 在后台按文章类型录入内容
3. 绑定分类、标签和专题信息
4. 发布后自动进入对应的专题聚合页
5. 前台首页和专题页展示对应入口

### 5.5 场景面试内容流程

1. 收集一个真实或高频面试场景
2. 拆解场景背景、问题点、错误方案和推荐方案
3. 形成结构化文章
4. 聚合到面试专题页
5. 在后续学习或面试复盘中持续更新

## 六、阶段性架构策略

### 6.1 第一阶段

策略：

- 单体架构优先
- 先做核心闭环
- 不做复杂权限系统
- 不做微服务拆分
- 搜索先使用数据库方案
- 首页先突出内容结构，不只展示最新文章列表

目标：

- 先把写文章和看文章链路打通
- 同时建立学习路线、学习总结、面试复盘、项目实战的内容入口

### 6.2 第二阶段

策略：

- 开始引入缓存和内容增强能力
- 对热点数据和阅读量做优化
- 增加图片上传、评论、点赞等互动模块

### 6.3 第三阶段

策略：

- 作为技术升级实验项目
- 逐步拆出异步处理、搜索和服务治理能力

## 七、面试表达价值

这个架构方案适合在面试中从以下角度展开：

- 为什么第一阶段选择单体架构
- 前后端为什么采用分离方式
- 为什么使用 JWT 做后台登录认证
- 为什么用 MyBatis-Plus 提升开发效率
- 如何从单体博客演进到 Redis、MQ、搜索和微服务

## 八、当前架构原则

- 先简单、后增强
- 先可运行、后优化
- 先单体、后扩展
- 先业务闭环、后技术亮点
- 先内容可沉淀、后展示可美化
